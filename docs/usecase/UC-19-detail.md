# UC-19 詳細 — 主家族が家族グループのアクセス権限・招待状態を管理する

## 1. ID / 名称 / 目的（価値）

- **ID**: UC-19
- **名称**: 主家族が家族グループのアクセス権限・招待状態を管理する
- **目的（価値）**: 家族グループ内のメンバー追加・削除・ロール変更・招待状態の可視化を主家族が一元的に行えるようにし、課題 I8（アクセス制御の不在）および I6（家族間タスク重複に伴う運用混乱）を構造的に解消する。家族外への情報漏えいリスクを抑えつつ、メンバー異動時にも安全に共有範囲を維持できる状態を提供する。

## 2. 主アクター

- **一次アクター**: 主家族メンバー（家族グループのオーナー / 管理者ロール）
- **二次アクター / 外部システム**:
  - 招待対象の家族メンバー（被招待者）
  - 認証基盤（サインアップ・サインインに利用、UC-01 と連携）
  - 通知配信チャネル（招待リンク送付のためのメール等。具体的なチャネル選定は `TBD`）

## 3. 前提条件

- 主家族メンバーが UC-01（サインアップ・ログイン）を完了し、認証済みセッションを保持している。
- UC-02（家族グループ作成・招待）により、対象の家族グループが少なくとも 1 件作成され、操作者が当該グループのオーナーまたは管理者ロールを保有している。
- 対象家族グループに紐づく犬プロフィール・記録データが存在しうる（メンバー削除時の影響評価のため）。

## 4. 基本フロー要約

1. 主家族メンバーが家族グループ管理画面を開く。
2. システムが当該グループの現メンバー一覧（氏名・ロール・参加日）と招待中（未承諾）の招待一覧（宛先・送信日・状態）を表示する。
3. 主家族メンバーが以下のいずれかの操作を選択する。
   - (a) 新規メンバーの招待: 宛先と付与ロールを指定し送信する。
   - (b) 既存メンバーのロール変更: 対象メンバーに対し新ロールを選択し確定する。
   - (c) メンバー削除: 対象メンバーを選択し削除を確定する（確認ダイアログ）。
   - (d) 招待の再送 / 取り消し: 招待中レコードに対し再送または取り消しを行う。
4. システムが権限チェック（操作者がオーナー / 管理者であること）を行ったうえで変更を永続化し、監査用に操作履歴を記録する。
5. システムが結果（成功 / 失敗）を画面に通知し、メンバー一覧および招待一覧を最新状態に更新する。

## 5. 主要例外

- **E1: 権限不足** → 操作者がオーナー / 管理者ロールでない場合、システムは操作を拒否し権限不足メッセージを表示する。
- **E2: 自分自身のオーナー権限剥奪 / 最後の管理者削除** → グループに管理者が 0 人になる操作はシステムが拒否し、先に別メンバーへ権限委譲するよう促す。
- **E3: 招待宛先の重複 / 既存メンバー** → 既に招待中、または既存メンバーである宛先への再招待はシステムが検知し、再送または取り消し動線へ誘導する。

## 6. 主要データ I/O

- **入力**:
  - 招待宛先識別子（メールアドレス等。形式は `TBD`）
  - 付与ロール（例: `owner` / `admin` / `member`。ロール体系は `TBD`、UC-02 と整合させる）
  - 対象メンバー ID（ロール変更・削除時）
  - 招待 ID（再送・取り消し時）
- **出力 / 永続化**:
  - 家族グループメンバーシップ（メンバー ID、ロール、参加日時、最終更新者）
  - 招待レコード（招待 ID、宛先、ロール、状態: 送信済 / 再送 / 取消 / 承諾、送信日時、有効期限）
  - 操作監査ログ（操作者、操作種別、対象、日時）

- **データ SoR / オーナー部門**: 家族グループ / メンバーシップ / 招待は本アプリ（家族健康管理 Web アプリ）が SoR。認証アカウント情報は認証基盤側 SoR（UC-01 と整合）。
- **依存システム**: 認証基盤、通知配信チャネル（招待リンク送付）。
